# Website Change Monitor MVP

Status: ready-for-agent

## Problem Statement

Нужно локальное однопользовательское приложение для Windows 11, которое регулярно проверяет выбранные части примерно ста общедоступных веб-страниц, устойчиво отличает содержательные изменения от нерелевантной разметки, сохраняет историю и сообщает пользователю о результате. Страницы могут отрисовываться JavaScript, а полезная информация может находиться в нескольких не смежных участках DOM и повторяемых карточках, поэтому одного корневого селектора недостаточно.

Пользователь должен видеть, что и когда проверялось, сравнивать последовательные состояния и понимать, дошло ли конкретное Уведомление до Telegram. Приложение должно оставаться работоспособным при недоступном Telegram, переживать перезапуски без потери очереди и данных и не требовать Docker, внешней базы данных или постоянно работающего сервера.

## Solution

Создать русскоязычное локальное веб-приложение, работающее одним foreground-процессом на Windows 11. Пользователь создаёт Мониторы с URL, одним или несколькими Целевыми селекторами, необязательными Селекторами исключения, Интервалом проверки и Метками. Перед сохранением приложение загружает страницу в изолированном Chromium, показывает совпадения каждого селектора и объединённую Целевую область в порядке DOM.

Один долговечный worker последовательно выполняет автоматические, Просроченные, Ручные и Повторные проверки. Первая успешная Проверка создаёт Базовый снимок. Следующие успешные Проверки формируют канонический Снимок и сравнивают его с предыдущим успешным состоянием; Изменение создаёт долговечное Уведомление и доступное из контекста Сравнение. Ошибка получает ровно одну Повторную проверку через минуту, и только повторная ошибка считается Окончательной ошибкой.

Интерфейс состоит из плотных таблиц «Мониторы», «Журнал» и «Уведомления» с верхней навигацией, правой панелью выбранного Монитора и широким модальным Сравнением в стиле Git-клиента. Браузерные уведомления и существующий Telegram-модуль получают события best effort. Недоступность Telegram не препятствует запуску и Проверкам: интерфейс и диагностика показывают деградированное состояние, а возле каждого предназначенного Telegram Уведомления отображается Состояние доставки Telegram.

React-интерфейс и агенты автоматизации используют один и тот же документированный локальный HTTP API. Контракт публикуется в машинно-читаемом OpenAPI 3.1 и в кратком руководстве с примерами ручных вызовов из PowerShell/curl; отдельного скрытого или привилегированного API для UI нет. Каждый вертикальный срез поставляет соответствующие бизнес-правила, хранение, HTTP-контракт, UI и тесты вместе, поэтому интерфейс не откладывается до завершения backend.

## User Stories

1. As a пользователь, I want to запустить приложение обычной командой или ярлыком на Windows 11 so that интерфейс открылся в браузере без Docker и ручного запуска нескольких сервисов.
2. As a пользователь, I want to повторно запустить уже работающее приложение so that открылось существующее окно, а не появился второй экземпляр.
3. As a пользователь, I want to создать Монитор с понятным именем и общедоступным HTTP(S) URL so that приложение начало наблюдать конкретную страницу.
4. As a пользователь, I want to указать несколько Целевых селекторов so that одна Целевая область могла включать разнесённые по странице элементы и повторяемые карточки.
5. As a пользователь, I want to видеть объединение совпадений Целевых селекторов без дублей и в порядке DOM so that Снимок отражал естественный порядок чтения страницы независимо от порядка полей формы.
6. As a пользователь, I want to указать несколько Селекторов исключения so that целые нерелевантные поддеревья, включая их текст и вложенные элементы, не попадали в наблюдение.
7. As a пользователь, I want to проверить синтаксис и совпадения каждого селектора до сохранения so that ошибочный или ничего не находящий Целевой селектор не запускал бесполезный Монитор.
8. As a пользователь, I want to видеть число совпадений каждого Целевого селектора и итоговый размер уникальной Целевой области so that было понятно, что именно будет наблюдаться.
9. As a пользователь, I want to выбрать Интервал проверки 6, 12, 24, 48 или 72 часа so that нагрузка и скорость обнаружения соответствовали важности страницы.
10. As a пользователь, I want to назначить Монитору несколько Меток и фильтровать по ним таблицу so that большой список Мониторов оставался управляемым.
11. As a пользователь, I want to получить Базовый снимок сразу после создания Монитора без Уведомления об Изменении so that последующие результаты имели корректную точку сравнения.
12. As a пользователь, I want to запустить Ручную проверку so that можно было проверить страницу немедленно, не дожидаясь срока.
13. As a пользователь, I want to приостановить автоматические Проверки, сохранив Историю Монитора и возможность Ручной проверки so that временная пауза не уничтожала данные.
14. As a пользователь, I want to возобновить Приостановленный монитор so that ожидающая Повторная или Просроченная проверка выполнилась перед обычным расписанием.
15. As a пользователь, I want to изменить имя, Интервал или Метки без потери истории so that организационные настройки не меняли смысл наблюдения.
16. As a пользователь, I want to получить явное предупреждение перед изменением URL или состава и значений селекторов so that я понимал, что новая Область наблюдения необратимо очистит прежнюю Историю Монитора.
17. As a пользователь, I want to переставить поля селекторов без очистки истории so that чисто визуальное упорядочивание формы не считалось сменой Области наблюдения.
18. As a пользователь, I want to удалить Монитор только после подтверждения его имени и предупреждения so that Монитор, его история и Снимки не были удалены случайно, а общие Метки сохранились.
19. As a пользователь, I want to видеть каждую состоявшуюся Проверку в Журнале so that можно было подтвердить работу расписания и исследовать ошибки.
20. As a пользователь, I want to получить ровно одну Повторную проверку через минуту после ошибки so that кратковременный сбой не создавал преждевременное Уведомление об Окончательной ошибке.
21. As a пользователь, I want to после простоя получить одну Просроченную проверку на Монитор без воспроизведения каждого пропущенного срока so that восстановление оставалось предсказуемым и не перегружало страницы.
22. As a пользователь, I want to видеть обнаруженное Изменение как добавление, удаление, перестановку структуры или изменение видимого текста so that значимые изменения не терялись, а классы, атрибуты и стили сами по себе не создавали шум.
23. As a пользователь, I want to открыть Сравнение из строки Проверки или Уведомления so that мне не приходилось знать или вводить внутренние идентификаторы.
24. As a пользователь, I want to изучить Сравнение в широком модальном окне с прежним состоянием слева, новым справа, красными удалениями и зелёными добавлениями so that различия читались как привычный Git diff.
25. As a пользователь, I want to видеть плотную таблицу Мониторов и правую панель с настройками, действиями и компактной Историей выбранного Монитора so that основная работа помещалась на одном экране без громоздких карточек и бокового меню.
26. As a пользователь, I want to получать Уведомления об Изменениях и Окончательных ошибках в интерфейсе so that важные результаты сохранялись независимо от внешних каналов.
27. As a пользователь, I want to явно разрешить браузерные системные уведомления so that открытая или фоновая вкладка могла сообщать о новых результатах без неожиданного запроса разрешения.
28. As a пользователь, I want to включить свитчер «Уведомлять при отсутствии изменений» в правой части шапки so that успешные Проверки без Изменений подтверждали штатную работу системы.
29. As a пользователь, I want to чтобы положение этого свитчера сохранялось между перезапусками so that Режим контрольных уведомлений не менялся самопроизвольно.
30. As a пользователь, I want to видеть постоянную заметную плашку под верхней навигацией, когда Telegram недоступен so that деградация канала была очевидна, но не мешала пользоваться приложением.
31. As a пользователь, I want to вручную перепроверить Telegram из плашки и перейти к безопасной диагностике so that можно было подтвердить восстановление без перезапуска приложения.
32. As a пользователь, I want to видеть возле каждого предназначенного Telegram Уведомления состояние «Отправляется», «Отправлено» или «Не отправлено» и безопасную причину so that результат доставки не смешивался с Результатом проверки.
33. As a пользователь, I want to чтобы старые неотправленные сообщения не догонялись после восстановления Telegram so that поздние и уже неактуальные уведомления не приходили неожиданно.
34. As a пользователь, I want to запускать диагностику перед стартом или обновлением so that фатальные проблемы отличались от необязательной недоступности Telegram.
35. As a пользователь, I want to создавать и проверять резервную копию остановленного приложения и восстанавливать её явной командой so that единственную локальную базу можно было безопасно защитить.
36. As a пользователь, I want to обновляться на точный Git tag с предварительной копией данных, проверками и парным откатом к старому коду и базе so that миграция не оставляла несовместимое состояние.
37. As a пользователь, I want to завершить приложение через Ctrl+C с коротким ожиданием текущей работы so that SQLite и история не повреждались при обычной остановке.
38. As an агент автоматизации, I want to использовать тот же локальный HTTP API, что и интерфейс, so that можно было создавать Мониторы, запускать Проверки и читать результаты без управления браузерным UI.
39. As a разработчик или оператор, I want to иметь актуальный OpenAPI-контракт и примеры PowerShell/curl so that доступные операции, форматы данных и ошибки можно было проверить без чтения реализации.

## Implementation Decisions

- Архитектура — модульный монолит на TypeScript в одном процессе Node.js 24 LTS: Fastify 5 предоставляет loopback-only HTTP/API и статический React/Vite UI, Playwright Library управляет одним долгоживущим bundled Chromium, а SQLite является единственным источником истины. В процессе работает ровно один scheduler worker и не более одной Проверки одновременно.
- Границы модулей: domain, application, scheduler, browser-playwright, persistence-sqlite, http-fastify, ui-react и notifications. Внешние зависимости подключаются через интерфейсы репозитория Мониторов, репозитория Проверок, исследования страницы, канала уведомлений, часов и очереди Проверок.
- Сервер слушает только `127.0.0.1` на фиксированном порту 43117, проверяет Host и Origin, не включает открытый CORS и не принимает соединения из LAN. Исследование страницы разрешает только публичные HTTP(S) адреса и повторно проверяет каждый redirect и DNS-результат, блокируя loopback, private, link-local, multicast и иные непубличные диапазоны.
- HTTP API является единым контрактом для React UI и локальной автоматизации. Он публикует OpenAPI 3.1, использует версионированные JSON-схемы и единый безопасный формат ошибок; руководство в репозитории показывает основные операции через PowerShell/curl. Новая пользовательская возможность не считается завершённой, пока её API-контракт, руководство и примеры не обновлены. Отсутствие browser Origin разрешается только для прямого loopback-клиента при корректном Host; браузерные запросы проходят строгую проверку Origin.
- Каждая Проверка получает новый непостоянный BrowserContext без учётных данных, разрешений, service workers, downloads и popups. Поддерживается только light DOM главного документа и стандартные CSS-селекторы; iframe, shadow DOM, XPath и Playwright-расширения селекторов исключены.
- Конвейер исследования ограничен 60 секундами: переход до `DOMContentLoaded`, проверка главного ответа и типа содержимого, ожидание непустого объединения целей, однократная прокрутка к первой отрисованной цели, одна секунда ожидания, затем окно тишины DOM 750 мс в пределах пяти секунд и атомарное синхронное извлечение. Viewport — 1440×900, scale 1, `ru-RU`, `Europe/Moscow`, светлая тема.
- Целевая область — уникальное объединение всех совпавших Целевых селекторов, глобально отсортированное в порядке DOM; один узел, найденный несколькими селекторами, включается один раз, а разные вложенные цели остаются отдельными. Пустое итоговое объединение — ошибка; исчезновение только одного селектора при непустом объединении после сохранения — обычное потенциальное Изменение.
- Исключения применяются внутри каждой корневой цели и удаляют совпавшие поддеревья целиком. Видимый текст берётся как `innerText` после временного скрытия Исключений и вместе со структурой извлекается одной атомарной операцией; временные изменения страницы обязательно восстанавливаются.
- Снимок — версионированный канонический JSON с упорядоченными целями, preorder-структурой элементов и нормализованными строками видимого текста. В структуру входят namespace, local name и число дочерних элементов, но не атрибуты, CSS-классы, стили, текстовые узлы или комментарии. Текст приводится к корректному Unicode, единым переводам строк и NFC, сохраняя значимые пробелы, NBSP, табы, регистр и пунктуацию.
- Канонические байты строятся по RFC 8785 JCS в UTF-8; равенство определяется байт-в-байт, SHA-256 служит только индексом. Неизвестная версия формата является технической ошибкой, а не Изменением. Лимиты Снимка: 1000 целей, 50 000 элементов, глубина 256, 20 000 строк текста и 8 MiB JSON.
- Сравнение использует детерминированный двухуровневый Myers diff с правилом удаления перед добавлением и без отдельного определения перемещений. Ограничения — 20 000 строк на секцию и 5 миллионов переходов графа; превышение даёт честный ограниченный результат или типизированную ошибку, но не ложное отсутствие Изменений.
- Долговечная SQLite-очередь хранит намерения `scheduled`, `overdue`, `manual` и `retry` со состояниями queued/running/finished/cancelled. Приоритет: Ручные, Повторные, Просроченные, плановые; после трёх Ручных обязательно выполняется старейшая автоматическая работа. Дубли Ручных запросов и работы уже выполняющегося Монитора объединяются.
- Следующий обычный срок вычисляется от завершения цепочки; Ручная проверка также сдвигает срок. Ошибка создаёт ровно одну Повторную проверку через минуту; её ошибка становится Окончательной. После перезапуска interrupted work фиксируется как ошибка, queued work сохраняется, а для каждого Монитора создаётся не более одной Просроченной проверки.
- SQLite работает в WAL с foreign keys, `synchronous=FULL`, busy timeout 5 секунд, явными транзакциями и нумерованными SQL-миграциями без ORM. Драйвер — `better-sqlite3`; релиз блокируется, если для поддерживаемой среды нет его prebuild. Требуется SQLite не ниже 3.51.3 либо версия с эквивалентными исправлениями.
- В одной базе хранятся Мониторы, отдельные упорядоченные записи Целевых селекторов и Селекторов исключения, Метки и связи, очередь, Проверки, Снимки, миграции, события Уведомлений и попытки доставки. Позиции селекторов нужны только UI; семантически это множества.
- Каждая начатая работа создаёт запись Проверки. Снимок хранится только для Базового снимка или Изменения; связи before/after позволяют вывести Сравнение без отдельной таблицы изменений. Данные всегда принадлежат конкретному Монитору и ревизии Области наблюдения.
- Подтверждённое изменение URL или множества селекторов атомарно повышает ревизию, удаляет прежнюю историю и очередь и назначает немедленную новую базу. Перестановка селекторов, имя, Интервал, Метки и пауза историю не меняют. Автоматическая очистка истории по давности в MVP отсутствует.
- События `change_detected`, `check_failed_final` и `control_check_ok` фиксируются в одной транзакции с Результатом проверки, Снимком и расписанием. Первые два видны в центре; контрольное событие видно в Журнале и Истории Монитора, но не загромождает центр. Значение режима читается в момент commit и не применяется задним числом.
- После commit события доставляются через SSE и Telegram. UI сначала читает REST high-water mark, затем SSE после event id; доставка at least once, клиент удаляет дубли. Replay после переподключения не создаёт повторных системных popups.
- Telegram использует установленный Python 3.11+ модуль `telegram-alert-bus` 0.1.5 через абсолютный путь к `telegram-alert.exe`. Процесс запускается напрямую без shell, получает строгий JSON через UTF-8 stdin и ограничен 70 секундами. Токен остаётся в Windows Credential Manager/keyring; приложение не реализует поверх модуля очередь, повторы или multipart-атомарность.
- Для каждого предназначенного Telegram события заранее создаётся запись доставки, даже если канал уже известен как недоступный. Старые pending при новом запуске становятся abandoned; unavailable, permanent, temporary, timeout и abandoned отображаются как «Не отправлено» и никогда не повторяются. При восстановлении отправляются только новые события.
- UI следует согласованному прототипу A: верхняя горизонтальная навигация без левого меню, плотные таблицы вместо крупных карточек, правая контекстная панель Монитора и широкое почти полноэкранное модальное Сравнение. Отдельного пункта навигации «Сравнение» и ручного ввода ID нет.
- Telegram проверяется при старте, по кнопке и перед новым предназначенным ему событием, если канал считался недоступным. Плашка под шапкой присутствует на всех основных экранах до восстановления. Недоступность Telegram соответствует деградированному, но рабочему состоянию приложения.
- Поддерживается только Windows 11 x64 и foreground source-install под обычным пользователем. Изменяемые данные размещаются под `%LOCALAPPDATA%/WebsiteChangeMonitor`; Chromium устанавливается локально приложению. `doctor` возвращает 0 для готовности, 1 для фатальной проблемы ядра и 2 для деградации Telegram; start/update принимают 0 и 2.
- Логи — редактированные NDJSON с ротацией 10 MiB и хранением 20 файлов. Backup/restore выполняются только при остановленном приложении через SQLite API с проверкой; ручные копии не удаляются, автоматические pre-update и pre-migration хранятся по три. Обновление выполняется на точный Git tag и допускает только парный откат к прежнему коду и pre-update базе после миграции.
- Корректное завершение перестаёт принимать новую работу, ждёт текущую не более восьми секунд, фиксирует прерывание и закрывает Chromium и SQLite.

## Testing Decisions

- Основной тестовый шов — application/use-case слой целиком с временной настоящей SQLite и подменами PageProbe, NotificationSink и Clock. На нём проверяются планирование, приоритеты и защита от голодания, дедупликация, единственная Повторная проверка, пауза/возобновление, восстановление после сбоя, очистка Области наблюдения и атомарность Check/Snapshot/Notification.
- Адаптер Playwright проверяется настоящим bundled Chromium на локальных fixture-страницах. Набор покрывает JavaScript rendering, естественный DOM-порядок нескольких целей, повторяемые и вложенные цели, дедупликацию узлов, несколько Исключений, видимый/скрытый текст, окно стабильности, исчезающие узлы, redirect/DNS-защиту, запрещённые адреса, таймауты и лимиты Снимка.
- Persistence-адаптер проверяется настоящей SQLite: миграции вперёд, ограничения и каскады, WAL/recovery, `quick_check`, `foreign_key_check`, конкурентная защита одного worker, очистка ревизии, backup/restore и парный rollback. Тесты должны доказывать инварианты, а не внутреннее устройство SQL-функций.
- Telegram-адаптер тестируется fake executable, который фиксирует UTF-8 stdin и моделирует exit codes, зависание, недоступность, временные и постоянные ошибки. Проверяются 70-секундный предел, отсутствие shell, неизменяемые Состояния доставки Telegram, abandoned после перезапуска и отсутствие доставки старых сообщений после восстановления. Отдельный ручной smoke подтверждает контракт с установленным реальным модулем без использования настоящего токена в автоматических тестах.
- UI/API проверяются на уровне пользовательских потоков: создание и preview с несколькими селекторами, подтверждение очистки истории, пауза и Ручная проверка, таблицы и правая История, контекстное открытие широкого Сравнения, режим «Уведомлять при отсутствии изменений», Telegram-плашка и состояния доставки, REST+SSE reconnection и явное разрешение браузерных уведомлений.
- OpenAPI проверяется на соответствие фактическим Fastify routes и JSON-схемам. Для каждого вертикального среза контрактные тесты вызывают новую возможность напрямую через HTTP без React UI, а отдельный smoke выполняет документированные PowerShell/curl-сценарии создания Монитора, запуска Ручной проверки и чтения результата.
- Релизный smoke выполняется в чистом профиле Windows 11 x64, включая пути с пробелами и кириллицей: `npm ci`, установка Chromium, build, doctor в ready и degraded режимах, первый/повторный start, один успешный baseline/change flow, Ctrl+C, backup/restore, update и rollback.
- Не закреплять тестами приватные helpers, случайные временные задержки, точную DOM-разметку React-компонентов или нестабильный внешний сайт. Детерминированные часы, fixture-страницы и контролируемые процессы являются обязательными на автоматическом контуре.

## Out of Scope

- Облачный или постоянно доступный сервер, многопользовательский режим, macOS, Linux, Windows 10, сервис Windows, Task Scheduler, автозапуск при входе, Electron/desktop packaging, Docker, Redis, внешняя база данных и микросервисы.
- Авторизованные страницы, хранение учётных данных страницы, CAPTCHA, обход антибот-защиты, клики, сценарии взаимодействия, кнопочная пагинация и бесконечная прокрутка.
- Интерактивный выбор DOM-элемента кликом; селекторы копируются через DevTools и проверяются preview.
- Iframe, shadow DOM, XPath, Playwright-specific selectors и исследование неосновного документа.
- Web Push и уведомления после закрытия вкладки; поддерживаются только открытая или фоновая вкладка текущего локального приложения.
- Доставка старых Telegram-сообщений задним числом, дополнительная очередь/брокер Telegram, собственные повторы поверх `telegram-alert-bus` и гарантированная атомарность multipart.
- Группы Мониторов, международный интерфейс и крупные dashboard-карточки.
- Автоматическое удаление текущей Истории Монитора по сроку хранения.
- Реализация приложения в рамках planning-этапов `/to-spec` и `/to-tickets`; утверждённая спецификация опубликована как отдельные tracer-bullet GitHub Issues, а выполнение начинается только отдельным вызовом `/implement`.
