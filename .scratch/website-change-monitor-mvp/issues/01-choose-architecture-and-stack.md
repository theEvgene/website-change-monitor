# Выбрать архитектуру и технологический стек

Type: research
Status: resolved
Blocked by: none

## Question

Какая минимальная архитектура и технологический стек лучше всего удовлетворяют подтверждённым требованиям локального Windows-веб-приложения: управляемый JavaScript-браузер, около 100 Мониторов, планировщик в серверном процессе, локальная база, русский UI и запуск одной командой? Сравнить реалистичные варианты, выбрать один и зафиксировать границы модулей и процесса.

## Answer

### Решение

Выбрать **модульный монолит на TypeScript в одном Node.js 24 LTS-процессе**:

- HTTP-сервер и API: **Fastify 5**, слушающий только `127.0.0.1`;
- UI: **React + Vite**, собранные статические файлы раздаёт тот же Fastify-процесс; UI сразу пишется по-русски, без слоя интернационализации в MVP;
- исследование страниц: **Playwright Library + поставляемый им headless Chromium**;
- локальная БД: **SQLite в WAL-режиме через `better-sqlite3`**, подготовленные выражения, явные транзакции и пронумерованные SQL-миграции без ORM;
- планирование: собственный прикладной планировщик поверх сохранённого в SQLite `next_check_at` и ограниченной очереди Проверок; системный cron, Windows Task Scheduler и отдельный брокер не нужны;
- запуск: один production entry point (`npm start`), который проверяет окружение, применяет миграции, поднимает HTTP-сервер и планировщик, запускает Chromium и открывает UI. Детали первоначальной установки, ярлыка и расположения данных остаются решению «Определить запуск и эксплуатацию на Windows».

Node.js 24 выбран как текущая LTS-линия, которую сам проект Node рекомендует для production-приложений; она поддерживается до апреля 2028 года ([Node.js Releases](https://nodejs.org/en/about/previous-releases), [переход v22 → v24](https://nodejs.org/en/blog/migrations/v22-to-v24)). Возможность оставить уже установленный Node.js 20.11.1 без обновления отдельно пересмотрена и отклонена: линия 20 стала EOL 24 марта 2026 года и больше не получает исправления безопасности ([Node.js EOL](https://nodejs.org/en/about/eol)); текущий Playwright поддерживает Node 22/24/26, а текущий Vite требует как минимум Node 20.19 или 22.12 ([Playwright system requirements](https://playwright.dev/docs/intro#system-requirements), [Vite Getting Started](https://vite.dev/guide/)). То есть `20.11.1` несовместим с актуальным инструментарием даже помимо EOL. Закреплять исторические версии Playwright/Vite ради отказа от одного обновления означало бы начать MVP на неподдерживаемом runtime и отказаться от будущих исправлений; такой компромисс не принимается.

Fastify даёт небольшой HTTP-каркас с TypeScript, JSON Schema-валидацией и изолируемыми плагинами, при этом по умолчанию слушает только loopback ([Fastify TypeScript](https://fastify.dev/docs/latest/Reference/TypeScript/), [Plugins](https://fastify.dev/docs/latest/Reference/Plugins/), [Getting Started](https://fastify.dev/docs/latest/Guides/Getting-Started/)). React оправдан не самим списком Мониторов, а интерактивными формами, состояниями выполняющихся Проверок и двухколоночным Сравнением; Vite штатно собирает статический production bundle, который можно раздавать существующим backend-сервером ([React: Describing the UI](https://react.dev/learn/describing-the-ui), [Vite: Building for Production](https://vite.dev/guide/build.html), [Vite: Backend Integration](https://vite.dev/guide/backend-integration.html)). Метафреймворк с SSR здесь не нужен.

### Модель процессов

В MVP есть ровно три исполняемых участника:

1. **Один основной Node.js-процесс** владеет HTTP API, прикладными сценариями, планировщиком, очередью, подключением к SQLite и адаптерами уведомлений. Он является единственным писателем в БД.
2. **Один долгоживущий Chromium-процесс**, запущенный и контролируемый Playwright как дочерний процесс. Для каждой Проверки создаётся новый непостоянный `BrowserContext` и страница, после Проверки контекст гарантированно закрывается. Такие контексты изолируют cookies/local storage и не пишут данные просмотра на диск ([Playwright BrowserContext](https://playwright.dev/docs/api/class-browsercontext)). При падении Chromium адаптер помечает текущие Проверки ошибкой и один раз перезапускает браузер; политика повтора самой Проверки принадлежит оркестратору.
3. **Обычная вкладка Edge/Chrome пользователя** отображает UI и получает события от локального сервера. Это не управляемый Chromium и не часть серверной надёжности: закрытие вкладки не останавливает Node.js и не отменяет Проверки.

Отдельные worker-процессы, Electron main/renderer, Docker, Redis и внешняя СУБД для масштаба примерно в 100 Мониторов не вводятся. Одновременно выполняются максимум 1–2 Проверки (начальное значение — 1); это ограничивает память Chromium и нагрузку на сайты. Увеличивать параллелизм можно только после измерения длительности и пиков памяти на реальном наборе страниц.

### Границы модулей

Границы логические, внутри одного пакета и процесса; это не микросервисы и не npm-workspace из множества пакетов.

- **`domain`** — термины и чистые правила: Монитор, Область наблюдения, Проверка, Снимок, Изменение, Сравнение, интервалы и Результат проверки. Не импортирует Fastify, Playwright, React или SQLite.
- **`application`** — сценарии `create/update/pause monitor`, `preview`, `run check now`, выполнение Просроченной проверки, построение журнала и выдача Сравнения. Здесь же находятся порты `MonitorRepository`, `CheckRepository`, `PageProbe`, `NotificationSink`, `Clock` и `CheckQueue`. Оркестратор решает порядок «загрузить → извлечь → сравнить → атомарно сохранить → уведомить», но не знает конкретных библиотек.
- **`scheduler`** — читает ближайшие сроки через репозиторий, ставит разблокированные Проверки в ограниченную очередь и держит только один ближайший таймер пробуждения. Источник истины — сохранённый срок, а не таймер: Node прямо предупреждает, что `setTimeout` не гарантирует точный момент вызова ([Node.js Timers](https://nodejs.org/api/timers.html)). После перезапуска модуль заново вычисляет фронт просроченных Проверок, поэтому простой не требует воспроизведения всех пропущенных интервалов.
- **`browser-playwright`** — единственное место, где видны типы Playwright. Владеет жизненным циклом Chromium/context/page, таймаутами, переходом по URL, ожиданием, однократной прокруткой и сериализацией результата исследования. Playwright официально поддерживает Windows и устанавливает привязанные к своей версии браузерные бинарники; для MVP достаточно одного Chromium (`npx playwright install chromium`) ([Playwright Library](https://playwright.dev/docs/library), [Browsers](https://playwright.dev/docs/browsers)). API `locator.scrollIntoViewIfNeeded()` и `locator.evaluateAll()` непосредственно покрывают прокрутку к цели и обработку всех совпавших элементов ([scrollIntoViewIfNeeded](https://playwright.dev/docs/api/class-locator#locator-scroll-into-view-if-needed), [evaluateAll](https://playwright.dev/docs/api/class-locator#locator-evaluate-all)). Точная модель ожидания и извлечения остаётся билету «Определить модель загрузки и извлечения страницы».
- **`persistence-sqlite`** — подключение, миграции и реализации репозиториев. Один файл БД хранит конфигурацию, сроки и журнал; крупные побочные артефакты допускаются только если последующее решение о хранении это обоснует. SQLite предназначен именно для локального хранилища отдельного приложения и не требует отдельного администратора/сервера ([Appropriate Uses for SQLite](https://www.sqlite.org/whentouse.html)). WAL допускает одновременное чтение и запись, но всё равно имеет только одного писателя, что совпадает с выбранным процессом ([SQLite WAL](https://www.sqlite.org/wal.html)). На открытии явно включаются `journal_mode=WAL`, `foreign_keys=ON` и конечный `busy_timeout`; записи одного Результата проверки и соответствующего нового срока делаются одной короткой транзакцией. Точная схема и политика истории остаются отдельному билету.
- **`notifications`** — fan-out через независимые адаптеры интерфейса, браузерной вкладки и Telegram. Контракт принимает уже зафиксированное доменное событие и не участвует в транзакции БД; конкретный Telegram-модуль подключается адаптером после изучения его контракта.
- **`http`** — Fastify routes, валидация входа, преобразование DTO и раздача production UI. Он вызывает только прикладные сценарии; маршруты не обращаются напрямую к SQLite или Playwright. Канал серверных событий в браузер остаётся портом до решения билета об уведомлениях.
- **`web`** — React-компоненты и клиент API. В UI не дублируются правила планирования, сравнения или очистки истории; русские подписи используют язык из `CONTEXT.md`.
- **`main`** — composition root и жизненный цикл: конфигурация, миграции, сборка адаптеров, запуск HTTP/Chromium/scheduler, обработка завершения процесса. Только здесь инфраструктурные реализации связываются с портами.

Зависимости направлены внутрь: `web/http/infrastructure → application → domain`. Это оставляет чистые алгоритмы Снимка и расписания тестируемыми без браузера и БД и позволяет заменить Telegram-адаптер, не меняя выполнение Проверки.

### Почему `better-sqlite3`, а не встроенный `node:sqlite`

Оба API синхронные, что приемлемо при коротких транзакциях и единственном локальном пользователе. Однако в Node.js 24 встроенный `node:sqlite` на момент решения всё ещё имеет статус **1.2 Release candidate**, то есть формально остаётся экспериментальным и не подпадает под обычные гарантии стабильности ([Node.js 24 `node:sqlite`](https://nodejs.org/download/release/latest-v24.x/docs/api/sqlite.html), [индекс стабильности Node](https://nodejs.org/download/release/latest-v24.x/docs/api/documentation.html)). Поэтому MVP фиксирует зрелый `better-sqlite3`: он поддерживает транзакции, рекомендует WAL и публикует готовые бинарники для LTS-версий Node ([официальный репозиторий](https://github.com/WiseLibs/better-sqlite3)).

Цена решения — native addon. В билете про Windows-запуск обязательно проверить чистую установку конкретных зафиксированных версий Node 24 x64 + `better-sqlite3` без Visual Studio Build Tools. Если готового бинарника для зафиксированной пары нет, допустимый локальный пересмотр — перейти на `node:sqlite` после отдельного compatibility test; архитектурные порты и SQL-миграции от этого не меняются.

### Рассмотренные альтернативы

- **Python + FastAPI + Playwright Python + `sqlite3` + React.** Технически жизнеспособно: у Playwright есть официальный async Python API и сценарий упаковки через PyInstaller ([Playwright Python Library](https://playwright.dev/python/docs/library)). Не выбрано, потому что UI всё равно остаётся TypeScript/JavaScript, появляются два менеджера зависимостей и две модели контрактов, а автоматизация/сервер/клиент больше не используют один язык. Вернуться к варианту стоит только если проверка существующего Telegram-модуля покажет жёсткую Python-зависимость, которую дороже адаптировать.
- **Electron + Node/React.** Electron уже встраивает Chromium и вводит main/renderer/preload + IPC и отдельные правила изоляции ([Electron Process Model](https://www.electronjs.org/docs/latest/tutorial/process-model)). Для исследования внешних страниц всё равно нужен изолированный управляемый браузер, а обязательные уведомления рассчитаны на открытую обычную вкладку. Значит, desktop shell добавляет второй Chromium и упаковочную поверхность, не закрывая требования MVP.
- **Tauri.** Уменьшает размер desktop shell, но добавляет Rust toolchain, WebView2 и IPC/sidecar-границы на Windows ([Tauri Prerequisites](https://v2.tauri.app/start/prerequisites/)). Нативная оболочка не является требованием, поэтому эта сложность не окупается.
- **PostgreSQL/Redis/очередь задач или отдельный scheduler service.** Они решают многопользовательскую/распределённую конкуренцию, которой здесь нет, и ломают запуск одной командой. SQLite прямо рекомендует client/server БД, когда одна база одновременно доступна множеству компьютеров по сети; это противоположно текущему случаю ([SQLite: Appropriate Uses](https://www.sqlite.org/whentouse.html)).
- **Puppeteer вместо Playwright.** Может выполнить базовую Chromium-автоматизацию и тоже поддерживает изолированные browser contexts ([Puppeteer BrowserContext](https://pptr.dev/api/puppeteer.browsercontext)). Playwright выбран из-за прямого совпадения его locator API с требованиями «все совпадения + прокрутка к цели», официальной установки версионированного Chromium и возможности позднее проверять тот же поток в нескольких движках без смены адаптера.
- **SSR/meta-framework или чистые шаблоны вместо React/Vite.** SSR не даёт преимущества локальному однопользовательскому приложению без SEO. Серверные шаблоны уменьшают стартовую зависимость, но усложняют интерактивное Сравнение и синхронизацию состояний Проверок. React/Vite остаётся небольшой отдельной UI-границей; глобальный state framework в MVP не добавляется.

### Риски и обязательные проверки следующих решений

- **Windows dependency smoke test:** на чистом Windows 11 x64 проверить фиксированные Node 24, `npm ci`, бинарник `better-sqlite3`, project-команду `npm run install:browsers`, production build/start и пути без прав администратора. Playwright сейчас официально указывает Windows 11+ в системных требованиях ([Installation](https://playwright.dev/docs/intro)); Windows 10 не следует обещать без отдельной проверки.
- **Ресурсы Chromium:** измерить 1 и 2 параллельные Проверки на тяжёлых JS-страницах; зафиксировать ограничение памяти и перезапуск браузера. До измерений concurrency=1.
- **Блокировка event loop:** синхронный SQLite допустим только для коротких запросов/транзакций; не выполнять в транзакции browser I/O, построение diff или отправку уведомлений. Если профилирование покажет паузы, переносится только SQLite-адаптер в worker thread, а не весь сервер.
- **Версия SQLite для WAL:** зафиксированная сборка `better-sqlite3` должна включать SQLite 3.51.3+ либо официальный исправленный backport. SQLite сообщает о редкой WAL-reset corruption в версиях до 3.51.2 при нескольких соединениях; это особенно важно проверить до возможного переноса адаптера в worker thread ([SQLite WAL-reset bug](https://sqlite.org/wal.html#the_wal_reset_bug), [SQLite 3.51.3](https://sqlite.org/releaselog/3_51_3.html)).
- **Надёжность расписания:** таймер — лишь сигнал пробуждения; сроки, число повторов и состояния Проверки должны атомарно жить в БД. Конкретные гонки ручной/автоматической Проверки решает билет о выполнении и планировании.
- **Локальная поверхность доступа:** слушать только `127.0.0.1`, проверять допустимые `Host`/`Origin`, не включать CORS `*` и не принимать удалённые подключения. Если позже понадобится LAN-доступ, это новое требование с аутентификацией, а не переключение host на `0.0.0.0`.
- **Версионная фиксация:** коммитить lockfile и фиксировать Node/Playwright/Chromium. Документация Playwright прямо указывает, что каждой версии нужны определённые browser binaries и после обновления может понадобиться повторная установка ([Browsers](https://playwright.dev/docs/browsers)).
- **Telegram-интеграция:** пока сохранять порт `NotificationSink` и не протаскивать типы неизвестного модуля в domain/application. Если модуль требует отдельного runtime, это будет узкий adapter/child-process, а не основание заранее делить весь монолит.

Итого: выбранный стек минимизирует число runtime и языков, оставляет расписание и историю восстанавливаемыми после простоя, использует отдельный управляемый Chromium для JavaScript-страниц и сохраняет ясные порты там, где последующие билеты ещё должны принять предметные решения.
