# Спроектировать выполнение и планирование Проверок

Type: research
Status: resolved
Blocked by: 01, 02

## Question

Как серверный процесс должен планировать, ставить в очередь и ограничивать параллельность Проверок примерно 100 Мониторов, реализуя заданные интервалы, ручной запуск, приостановку, одну Просроченную проверку, повтор через минуту, таймауты и корректное завершение на Windows?

## Answer

### Решение

Использовать **долговечную очередь в SQLite и один in-process worker**. SQLite — источник истины для сроков и состояний; память содержит только ближайший таймер пробуждения и текущую выполняемую Проверку. `setTimeout` не является расписанием: Node.js не гарантирует точный момент или порядок срабатывания таймеров ([Node.js Timers](https://nodejs.org/api/timers.html#scheduling-timers)). После каждого пробуждения процесс заново читает готовую работу из БД.

Начальная глобальная параллельность — **1 Проверка**. Для примерно 100 Мониторов это ограничивает память Chromium и нагрузку на страницы, а минимальный Интервал проверки в шесть часов оставляет достаточную пропускную способность. Переход к 2 допускается только после измерений билета «Определить запуск и эксплуатацию на Windows»; модель очереди от этого не меняется. Для одного Монитора в любом случае разрешена не более чем одна queued/running Проверка.

### Долговечная модель работы

Техническая запись очереди (`CheckIntent`) содержит `monitor_id`, `kind`, `ready_at`, монотонный `sequence`, `chain_id`, `attempt_no` (`1` или `2`) и `scope_revision`. Состояния: `queued → running → finished`; отменённая до старта запись становится `cancelled`. Это инфраструктурные состояния, не новые доменные термины.

`kind` принимает `scheduled`, `overdue`, `manual` или `retry`. Каждая фактически начатая запись создаёт отдельную Проверку в журнале. Поэтому Повторная проверка — вторая Проверка, связанная с исходной через `chain_id`, а не скрытая попытка внутри одного результата.

Ограничения БД гарантируют не более одной активной записи (`queued` или `running`) на Монитор и не более двух Проверок в цепочке. Claim следующей записи выполняется короткой SQLite-транзакцией: выбрать допустимую queued-запись, перепроверить состояние/ревизию Монитора, перевести её в `running` и создать строку журнала. Завершение также атомарно фиксирует Результат проверки, новый срок или Повторную проверку и освобождает активную запись. SQLite начинает неявную транзакцию перед изменением и делает изменения видимыми атомарно при commit ([SQLite Transactions](https://www.sqlite.org/lang_transaction.html), [Isolation](https://www.sqlite.org/isolation.html)).

### Сроки и обычное планирование

- Сроки хранятся как UTC instants (`next_due_at`), а не как cron-выражения. Интервал берётся из фиксированного набора 6/12/24/48/72 часа.
- Новый подтверждённый Монитор получает `next_due_at = now`, чтобы первая успешная Проверка как можно раньше создала Базовый снимок.
- После успешной обычной или Ручной проверки новый срок равен `completed_at + текущий Интервал проверки`.
- После ошибки первой Проверки обычный срок временно не назначается: создаётся ровно одна Повторная проверка с `ready_at = completed_at + 1 минута`.
- После завершения Повторной проверки — успешно или с ошибкой — назначается `next_due_at = completed_at + текущий Интервал проверки`; третьей Проверки в цепочке нет.
- Если Интервал проверки изменён, пока Проверка стоит в очереди или выполняется, следующий срок рассчитывается по **новому** значению, прочитанному в завершающей транзакции.

Таким образом, Ручная проверка действительно сдвигает расписание. Если она завершилась ошибкой, окончательная точка сдвига — завершение её Повторной проверки. Ошибка не создаёт одновременно retry и обычную Проверку.

Scheduler держит один таймер до `min(ближайший ready_at/next_due_at, now + 60 секунд)`. Ограничение сна в 60 секунд позволяет заметить перевод часов и изменения из API. По пробуждению одной транзакцией материализуются все наступившие сроки, для которых нет активной работы, но worker забирает только одну запись. Сдвиг часов вперёд делает просроченные сроки готовыми один раз; сдвиг назад не дублирует работу. Для wall-clock сроков используется UTC-время, а для 60/75-секундных runtime deadlines — монотонный `performance.now()`.

### Просроченная проверка и запуск приложения

Старт процесса выполняет одну recovery-транзакцию до запуска HTTP API и worker:

1. Каждая оставшаяся `running` запись предыдущего процесса завершается ошибкой `process_interrupted` на время старта. Для `attempt_no = 1` создаётся одна Повторная проверка со сроком через минуту; после прерванной Повторной проверки назначается следующий обычный срок без третьей попытки.
2. Существующие queued-записи сохраняются; дубли по Монитору невозможны из-за ограничения БД.
3. Для каждого активного Монитора без queued/running работы и с `next_due_at <= startup_at` создаётся **ровно одна** Просроченная проверка. Пропущенные интервалы не воспроизводятся, потому что у Монитора хранится один срок, а не серия cron occurrences.
4. После commit запускаются worker и scheduler.

Если приложение не работало в момент срока Повторной проверки, сохраняется именно одна готовая Повторная проверка; отдельная Просроченная проверка для того же Монитора не создаётся. Следующий обычный срок появится после завершения цепочки.

### Очередность и объединение запросов

Детерминированный приоритет: `manual → retry → overdue → scheduled`, затем `ready_at`, затем `sequence`. Чтобы серия нажатий не вытеснила автоматическую работу навсегда, после трёх подряд Ручных проверок worker при наличии готовой автоматической записи обязан взять самую старую автоматическую.

Повторные команды и гонки объединяются:

- Ручная проверка без активной работы создаёт `manual` с `ready_at = now`.
- Если queued `scheduled/overdue` уже существует, она повышается до `manual` и становится готовой сейчас; второй записи нет.
- Если queued `retry` уже существует, команда «Запустить сейчас» лишь ускоряет её до `now`; `attempt_no = 2` сохраняется, поэтому новая команда не открывает бесконечную цепочку повторов.
- Если Проверка уже `running`, API возвращает её идентификатор и состояние «уже выполняется», не ставя ещё одну.
- Если автоматический срок наступил во время queued/running работы, новая запись не создаётся. Завершение текущей цепочки назначит один следующий срок.

У API-команды есть idempotency key, а уникальная активная запись остаётся последней защитой от двойного клика, повторной доставки HTTP-запроса и одновременного scheduler wake-up.

### Приостановка и изменение Монитора

Приостановленный монитор не выбирается scheduler и worker для автоматической, Просроченной или Повторной проверки, но Ручная проверка разрешена. Уже выполняющаяся Проверка при приостановке спокойно завершается: её принудительная отмена создала бы ложную ошибку. Queued автоматическая работа остаётся долговечной, но недоступной для выполнения.

При возобновлении сначала становится готовой сохранённая Повторная проверка, если она есть; иначе при прошедшем `next_due_at` создаётся одна обычная Проверка после возобновления без воспроизведения интервалов. Если срок ещё впереди, сохраняется прежний срок. Ручная проверка приостановленного Монитора обновляет будущий `next_due_at`, но не возобновляет автоматическую работу; её Повторная проверка также ждёт возобновления.

Каждое изменение Области наблюдения увеличивает `scope_revision`, удаляет queued работу старой ревизии и запрашивает отмену running `PageProbe`. Перед стартом и перед commit результата worker сравнивает захваченную ревизию с текущей. Устаревшее наблюдение отбрасывается и не может воскресить очищенную историю; для новой Области наблюдения создаётся новая немедленная Проверка для Базового снимка. Изменения остальных настроек ревизию не меняют.

### Таймауты, ошибки и уведомления

У `PageProbe` остаётся жёсткий deadline 60 секунд из билета «Определить модель загрузки и извлечения страницы». На весь orchestration одной Проверки устанавливается deadline **75 секунд**: подготовка, `PageProbe`, построение/сравнение Снимка, атомарный commit и cleanup. Он реализуется через `AbortController`; Node предоставляет стабильный `AbortSignal.timeout()` и композицию сигналов ([Node.js AbortController](https://nodejs.org/api/globals.html#class-abortcontroller)). Проигравший `Promise.race` без реальной отмены недопустим.

Любая ошибка первой Проверки — navigation, target, timeout, snapshot, browser crash или interruption процесса — создаёт одну Повторную проверку. Ошибка Повторной проверки завершает цепочку. Каждый начатый результат попадает в журнал; queued/cancelled до старта запись Проверкой не считается.

Завершающая транзакция сначала фиксирует журнал, Снимок/срок/повтор и доменное событие, и только после commit событие передаётся адаптерам уведомлений. Конкретная гарантия доставки и outbox остаются билету «Спроектировать доставку уведомлений»; ошибка отправки уведомления не превращает уже успешную Проверку в ошибочную и не запускает её повторно.

### Корректное завершение на Windows

При явной остановке, `SIGINT`, `SIGBREAK`, `SIGTERM` или `SIGHUP` процесс один раз переходит в `draining`:

1. перестаёт принимать новые Ручные проверки и новые scheduler claims;
2. очищает таймер пробуждения и закрывает HTTP listener;
3. abort-сигналом останавливает текущий `PageProbe` и закрывает его `BrowserContext`;
4. в пределах **8 секунд** фиксирует `application_shutdown` как ошибку начатой Проверки, создаёт допустимую Повторную проверку, сбрасывает буферы логов и закрывает SQLite/Chromium;
5. завершает процесс без прямого `process.exit()` до окончания cleanup.

Node указывает, что на Windows закрытие консоли порождает `SIGHUP`, после которого ОС может безусловно завершить процесс примерно через 10 секунд; `SIGINT` поддерживается, а `SIGTERM` можно слушать, хотя он не является нативным Windows-сигналом ([Node.js Process signals](https://nodejs.org/api/process.html#signal-events)). Поэтому graceful shutdown — оптимизация, а не основа корректности: power loss, Task Manager kill или исчерпание восьми секунд оставят `running`, которое надёжно восстановит startup-транзакция.

### Проверенные гонки

- Два scheduler wake-up и ручной клик не создают две Проверки благодаря транзакционному claim и уникальной активной записи.
- Ручная проверка во время running не копится следом и не вызывает вечный сдвиг расписания.
- Ошибка, shutdown и повторный старт не могут породить больше одной Повторной проверки из-за `chain_id/attempt_no` и уникального ограничения.
- Сто пропущенных интервалов дают одну Просроченную проверку.
- Приостановка между ошибкой и retry блокирует автоматический старт, но не теряет Повторную проверку.
- Изменение Области наблюдения во время браузерной работы не записывает результат старой ревизии.
- Commit результата до crash считается завершённым; crash до commit восстанавливается как interrupted. Полусохранённого Результата проверки нет.

Итого: SQLite хранит намерения и сроки, in-memory timer лишь будит один worker, а все решения о duplicate/retry/overdue принимаются атомарно. Это даёт предсказуемую очередь для 100 Мониторов, ровно один повтор и восстановление после любого способа завершения Windows-процесса.
