### Установка, настройка и фактическая команда

Установка на Windows из корня модуля:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\install.ps1
```

Скрипт находит Python 3.11+, при отсутствии пытается установить Python 3.13 через `winget`, создаёт `.venv` и устанавливает пакет. Он не добавляет executable в `PATH`, поэтому надёжная команда интеграции — **абсолютный путь** к `.venv\Scripts\telegram-alert.exe`, а не bare `telegram-alert`:

```text
C:\Users\rb199787\Documents\repositories\telegram-alert-bus\.venv\Scripts\telegram-alert.exe send
```

Node.js должен запускать executable напрямую без shell, передавать один JSON в `stdin`, закрывать поток и интерпретировать exit code. Для русского текста в окружении child process обязательно установить `PYTHONUTF8=1` (или эквивалентный `PYTHONIOENCODING=utf-8`): проверенный Python при redirected stdin по умолчанию сообщил `cp1252`, тогда как Telegram Bot API требует UTF-8 ([Telegram Bot API: Making requests](https://core.telegram.org/bots/api#making-requests)).

Первоначальная интерактивная настройка:

```powershell
.\.venv\Scripts\telegram-alert.exe configure --device "home-pc" --chat-id "123456789"
```

Несекретные `device` и `chat_id` сохраняются в `%APPDATA%\telegram-alert-bus\config.json`; путь можно переопределить переменной `TELEGRAM_ALERT_CONFIG`. Bot token хранится через Python `keyring` с service `telegram-alert-bus` и key `telegram-bot-token`; на этой Windows-машине фактически выбран `keyring.backends.Windows.WinVaultKeyring`, то есть Windows Credential Manager. Интеграция не должна использовать `--bot-token`: этот аргумент обходит keyring и раскрывает секрет в командной строке процесса.

### Входной JSON и текст сообщения

Sender принимает объект **ровно с четырьмя полями**; пропуски и дополнительные поля отклоняются:

```json
{
  "monitor_id": "web-updates",
  "status": "warning",
  "observed_at": "2026-07-02T18:30:00+03:00",
  "message": "Обнаружено изменение."
}
```

- `monitor_id`: непустая строка после trim, максимум 100 символов;
- `status`: `info | success | warning | error`;
- `observed_at`: ISO 8601 с обязательным часовым поясом, максимум 64 символа;
- `message`: непустой plain text после trim, максимум 100 000 символов.

`device` во входе запрещён: он берётся из локальной конфигурации. Формат каждой первой/единственной части:

```text
<иконка> <monitor_id> · <device>
HH:mm dd.MM.yyyy

<message>
```

Время всегда переводится в фиксированный UTC+3. Статусы отображаются как `ℹ️`, `🟢`, `🟠`, `🔴`. `parse_mode` не передаётся, поэтому это plain text без Markdown/HTML-экранирования; preview ссылок отключён. Telegram ограничивает `sendMessage` 4096 символами ([официальный контракт `sendMessage`](https://core.telegram.org/bots/api#sendmessage)); модуль резервирует 16 символов, делит длинный текст по переводу строки/пробелу либо жёсткой границе и добавляет `[i/n] `.

### Ошибки, повторы и exit codes

Для **каждой части** выполняется до трёх HTTP-попыток с timeout 20 секунд: исходная, затем повторы после 1 и 3 секунд. HTTP `429`, `5xx`, сетевые ошибки, timeout и невалидный ответ считаются временными; `retry_after` Telegram имеет приоритет и в коде не ограничен общим deadline. Остальные HTTP-ошибки и ответ `ok: false` считаются постоянными. Такое чтение `ok/description/parameters` соответствует официальному Bot API ([формат ответа Telegram](https://core.telegram.org/bots/api#making-requests)).

Контракт процесса:

| Exit code | Значение |
| ---: | --- |
| `0` | все части отправлены |
| `2` | невалидный JSON или поля |
| `3` | отсутствует/повреждена конфигурация или token |
| `4` | постоянный отказ Telegram |
| `5` | временная ошибка после исчерпания повторов |
| `10` | неожиданная внутренняя ошибка |

Успех печатается в `stdout` как человекочитаемая строка с количеством частей; ожидаемые ошибки — в `stderr`. Машинный результат определяется exit code, парсить текст stdout/stderr как стабильный протокол нельзя.

### Ограничения для проектирования уведомлений

- В модуле **нет очереди, архива, идемпотентности и дедупликации**. После окончательной ошибки он теряет алёрт; решение о долговечности принадлежит вызывающему приложению.
- Multipart-отправка не атомарна: если первые части отправлены, а последующая завершилась ошибкой, повтор всего алёрта продублирует уже доставленные части.
- Повторы выполняются отдельно для каждой части; общего deadline нет, а `retry_after` может заметно продлить жизнь child process. Родитель должен иметь собственную политику ожидания и завершения процесса.
- Конфигурация задаёт ровно один `chat_id` и одно имя устройства. Keyring key также один на пользователя Windows; несколько Telegram-направлений модуль не моделирует.
- `monitor_id` выводится пользователю без преобразования. Итоговое решение о доставке передаёт туда понятное имя Монитора, а события сопоставляет статусам `warning` для Изменения, `error` для Окончательной ошибки и `success` для успешной Проверки без Изменений; доступный `info` в MVP не используется.
- Sender не принимает HTML, screenshot, готовый Снимок или структурированный diff; вся содержательная сводка должна быть заранее уложена приложением в `message`.
- Инсталляция добавляет отдельный Python runtime и привязывает запуск к местоположению `.venv`; проверка установки/переноса и окончательный путь executable должны войти в решение «Определить запуск и эксплуатацию на Windows».
