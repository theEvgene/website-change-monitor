# Спроектировать доставку уведомлений

Type: research
Status: resolved
Blocked by: 01, 04, 07

## Question

Как реализовать единый поток событий для центра уведомлений, Notifications API открытой или фоновой вкладки и существующего Telegram-модуля, включая Изменения, окончательные ошибки и глобальный Режим контрольных уведомлений, без Web Push и очереди сообщений после остановки сервера?

## Answer

### Решение и события

Выбрать **durable центр уведомлений с best-effort внешней доставкой**. Единственный источник истины — неизменяемая запись `notification_events` в той же SQLite, а SSE, Notifications API и Telegram — независимые проекции этой записи. Недоступность Telegram не блокирует запуск приложения, Проверки, центр или браузерные уведомления. WebSocket и service worker не нужны: сервер только отправляет события открытым вкладкам, для чего стандартный `EventSource` уже предоставляет UTF-8-поток, автоматическое переподключение и `Last-Event-ID` ([WHATWG Server-sent events](https://html.spec.whatwg.org/multipage/server-sent-events.html)). Закрытая вкладка ничего не получает; Web Push и отложенная доставка наружу после следующего запуска приложения отсутствуют.

Создаются ровно три вида события:

| `kind` | Когда создаётся | Центр | Telegram `status` |
| --- | --- | :---: | --- |
| `change_detected` | успешная Проверка обнаружила Изменение; Базовый снимок исключён | да | `warning` |
| `check_failed_final` | Повторная проверка завершилась Окончательной ошибкой | да | `error` |
| `control_check_ok` | не первая успешная Проверка завершилась без Изменений и в момент commit включён Режим контрольных уведомлений | нет | `success` |

Первый ошибочный результат, для которого назначена Повторная проверка, Уведомления не создаёт. Режим контрольных уведомлений управляется свитчером «Уведомлять при отсутствии изменений» справа в верхней шапке, сохраняется после перезапуска и действует до ручного выключения; автоматического срока нет. Его значение считывается в транзакции завершения Проверки: последующее включение/выключение ничего не создаёт и не отменяет задним числом. В центр попадают только Изменения и Окончательные ошибки; контрольный успех остаётся в журнале Проверок и отправляется лишь в живую вкладку и Telegram, как задано картой. `read_at` влияет только на счётчик непрочитанных, но не на Telegram или системное уведомление.

Запись хранит монотонный `id`, `kind`, `center_visible`, `monitor_id`, `scope_revision`, `check_id`/`chain_id`, `created_at`, готовые русские `title` и `body`, локальный `target_path` и уникальный `dedupe_key`: `change:<check_id>`, `final-error:<chain_id>` либо `control:<check_id>`. Текст сохраняется копией, поэтому переименование Монитора не переписывает историю. Уникальный индекс на `dedupe_key` делает повторный commit/recovery идемпотентным. Изменение Области наблюдения и удаление Монитора каскадно удаляют связанные события вместе с Историей Монитора, поэтому центр не сохраняет ссылок на уже удалённые Проверки и Сравнения.

### Транзакционная граница и отсутствие посмертной очереди

В одной короткой транзакции SQLite фиксируются Результат проверки, Снимок при необходимости, следующий срок/завершение цепочки и `notification_events`; для каждого события, предназначенного Telegram, одновременно вставляется единственная строка `notification_deliveries(event_id, channel, boot_id, state, failure_reason)` с `UNIQUE(event_id, channel)`. Строка создаётся и при уже известной недоступности канала, чтобы интерфейс мог показать «Не отправлено», а не отсутствие данных. Только **после commit** процесс публикует SSE и при доступном канале будит последовательный Telegram-dispatcher. Поэтому центр никогда не показывает событие без завершённой Проверки, а внешний канал никогда не опережает commit.

`notification_events` — история центра, но не внешняя очередь. Dispatcher берёт только строки со своим случайным `boot_id`. При штатной остановке он прекращает приём, даёт текущему child process ограниченное время и помечает оставшиеся `pending` как `abandoned`; при старте любые `pending/sending` прошлого `boot_id` также становятся `abandoned` и **не отправляются**. Недоступные, неуспешные и `abandoned` доставки остаются окончательно «Не отправлено»: восстановление Telegram не ставит их обратно в очередь. Следствие принято осознанно: crash или недоступность канала между commit и фактической отправкой может оставить событие только в интерфейсе. Это соответствует локальному продукту без постоянно доступного сервера и исключает запоздалые Telegram/Windows-уведомления после перезапуска. Ошибка доставки записывается в строке канала, но не порождает новое уведомление и не создаёт рекурсию.

### Центр и SSE replay

При загрузке UI делает `GET /api/notifications`, получает страницу записей с `center_visible = true` и `high_water_id`, отображает центр без системных всплытий, затем открывает `EventSource('/api/notifications/stream?after=<high_water_id>')`. SSE отправляет `event: notification`, `id: <notification_events.id>` и JSON записи; heartbeat-комментарий удерживает соединение. Это закрывает гонку между REST и подключением: сервер выбирает все более новые видимые для центра записи из SQLite, а контрольные события передаёт только как live-события текущего процесса.

При кратком разрыве браузер автоматически передаёт последний `id` как `Last-Event-ID`, который имеет приоритет над исходным query cursor, а сервер повторяет все более новые записи центра. SSE поэтому имеет семантику **at least once**, а клиент применяет событие идемпотентно по числовому `id`; назначение поля `id` и заголовка reconnect определено HTML Standard ([формат и `Last-Event-ID`](https://html.spec.whatwg.org/multipage/server-sent-events.html#the-last-event-id-header)). При неизвестном/слишком старом cursor сервер посылает `event: reset`, после чего UI заново читает REST. Catch-up и новая загрузка обновляют только центр; системное Notifications API сообщение создаётся исключительно для события, полученного live в текущем соединении, поэтому replay не порождает ретроактивных всплытий.

### Состояние доставки Telegram в интерфейсе

Каждое предназначенное Telegram Уведомление получает видимое Состояние доставки Telegram. В центре Уведомлений оно показывается возле Изменения или Окончательной ошибки; контрольный успех не входит в центр, поэтому его состояние показывается возле соответствующей Проверки в Журнале и Истории Монитора. Технические состояния сворачиваются в три пользовательских:

- `pending/sending` — «Отправляется»;
- `delivered` — «Отправлено»;
- `unavailable`, постоянный/временный отказ, `timeout` или `abandoned` — «Не отправлено» с короткой безопасной причиной.

У старого состояния «Не отправлено» нет команды повторной отправки. После восстановления канала только новые Уведомления могут перейти в «Отправляется» и «Отправлено».

### Открытая и фоновая вкладка

Видимые для центра события обновляют его всегда. При `document.visibilityState === 'visible'` любое live-событие показывается внутри UI как toast, а Изменение или Окончательная ошибка дополнительно увеличивает счётчик центра; системное уведомление не дублируется. При `hidden` вкладка создаёт непостоянный `new Notification(...)`; HTML определяет `visible/hidden` и событие `visibilitychange` именно для фоновой/скрытой страницы ([WHATWG Page visibility](https://html.spec.whatwg.org/multipage/interaction.html#page-visibility)). `tag: "wcm:<event_id>"` и `renotify: false` дают одинаковому событию из двух вкладок один стабильный ключ: Notifications Standard заменяет уведомление с тем же непустым tag и origin ([replacement по `tag`](https://notifications.spec.whatwg.org/#showing-a-notification)). Обработчик `click` фокусирует существующую вкладку и переходит на `target_path`; если вкладка закрыта, непостоянной доставки нет.

Разрешение спрашивается **только** по явной кнопке «Включить браузерные уведомления», а не при загрузке и не при первом событии. Состояния трактуются буквально: `default` — показать ненавязчивое предложение, `denied` — не спрашивать снова и дать инструкцию изменить разрешение браузера, `granted` — разрешить системные сообщения; центр и Telegram от этого не зависят. Notifications Standard задаёт эти три состояния, а MDN требует вызывать запрос в ответ на действие пользователя ([WHATWG permission](https://notifications.spec.whatwg.org/#permissions), [MDN `requestPermission()`](https://developer.mozilla.org/en-US/docs/Web/API/Notification/requestPermission_static)). `127.0.0.1` является potentially trustworthy origin, поэтому локальный HTTP может быть secure context ([Secure Contexts §3.1](https://www.w3.org/TR/secure-contexts/#is-origin-trustworthy)); всё равно проверяются `isSecureContext`, наличие `Notification` и ошибки конструктора. На `focus`/`visibilitychange` значение permission перечитывается, потому что пользователь может изменить его вне приложения.

### Telegram-адаптер

Использовать неизменённый `telegram-alert-bus` по проверенному решению № 7: один child process на событие, абсолютный путь к executable, запуск напрямую без shell, строгий JSON через закрываемый UTF-8 stdin, `PYTHONUTF8=1`, token только из Windows Credential Manager. В `monitor_id` передаётся понятное имя Монитора (NFC, не более 100 символов; при пустом/длинном имени — стабильное сокращение), `observed_at` содержит timezone, а `message` — plain-text вид события, URL и короткая безопасная сводка без сырого Снимка и секретов.

Доступность адаптера проверяется при старте, по явной кнопке «Проверить снова» на предупреждающей плашке и перед новым предназначенным Telegram Уведомлением, если канал отмечен недоступным. При успешной проверке плашка исчезает, и это новое Уведомление можно отправить; предыдущие неуспешные доставки не поднимаются. Пока канал недоступен, под верхней навигацией на всех основных экранах показывается заметная постоянная плашка «Telegram недоступен» с безопасной причиной, путём к диагностике и кнопкой повторной проверки.

Dispatcher сериализован: одновременно работает не более одного sender. Приложение ограничивает `message` 3000 Unicode code points, чтобы с заголовком модуля заведомо остаться одной частью и не входить в неатомарный multipart; Telegram допускает 4096 символов в `sendMessage` ([Bot API](https://core.telegram.org/bots/api#sendmessage)). Для child process действует общий hard deadline **70 секунд**: это покрывает штатные три HTTP-попытки модуля по 20 секунд с паузами 1 и 3 секунды, но ограничивает неограниченный `retry_after`; затем процесс завершается, а доставка получает `timeout`.

Повторно запускать sender на уровне приложения нельзя: Telegram не даёт idempotency key, а потерянный ответ или multipart может привести к дублю. Встроенные повторы модуля — единственные. Exit `0` означает `delivered`; `2/3/4` — постоянный отказ, `5` — исчерпанная временная ошибка, `10`, spawn error и deadline — технический отказ. `stdout/stderr` ограничиваются и сохраняются только как безопасная диагностика; машинное решение принимается исключительно по exit code. Фактические ограничения подтверждены исходниками [`telegram-alert-bus` 0.1.5](https://github.com/theEvgene/telegram-alert-bus/tree/3eafff87c1cceb040842d701928b2d492d27c45c/src/telegram_alert_bus) и официальным форматом ответов Telegram ([Bot API: Making requests](https://core.telegram.org/bots/api#making-requests)).

Итого: атомарная запись даёт полный и дедуплицированный центр, SSE безопасно восстанавливает UI после reconnect, а состояние каждой Telegram-доставки остаётся видимым. Notifications API и Telegram намеренно best-effort только в течение жизни текущего серверного процесса — без блокировки приложения, Web Push, ретроактивных всплытий и посмертной очереди.
