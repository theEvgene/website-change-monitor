# Определить запуск и эксплуатацию на Windows

Type: research
Status: resolved
Blocked by: 01, 04, 05

## Question

Какой минимальный воспроизводимый способ установки зависимостей, первого запуска, повторного ручного запуска, открытия веб-интерфейса, хранения данных, диагностики и безопасного обновления подходит пользователю Windows без опыта Docker, не добавляя автозапуск или настольную упаковку?

## Research

- [Исследование запуска и эксплуатации на Windows](../research/09-windows-launch-and-operations.md)

## Answer

### Поддерживаемая среда

MVP поставляется как исходная установка для обычной учётной записи **Windows 11 x64** и работает одним foreground-процессом. Поддерживаемая версия среды — последний исправленный выпуск **Node.js 24 LTS x64**; `package.json` ограничивает `engines.node` диапазоном `>=24 <25`, а `doctor` и production entry point завершаются ошибкой на другом major.

На проверенной машине установлен Node.js 20.11.1. Оставить его без обновления нельзя: Node 20 завершил поддержку 24 марта 2026 года, текущий Playwright поддерживает только Node 22/24/26, а текущему Vite внутри ветки 20 требуется минимум 20.19 ([Node.js Releases](https://nodejs.org/en/about/previous-releases), [Playwright system requirements](https://playwright.dev/docs/intro#system-requirements), [Vite Getting Started](https://vite.dev/guide/)). Поэтому перед первым запуском требуется обновление до Node 24; фиксация старых Playwright/Vite ради `20.11.1` не является поддерживаемым вариантом MVP.

Версии JavaScript-зависимостей фиксирует committed `package-lock.json`. Установка выполняется только через `npm ci`; отдельно устанавливается только Chromium, соответствующий зафиксированной версии Playwright. Зафиксированная пара Node 24/`better-sqlite3` должна иметь готовый Windows x64 prebuild: Visual Studio Build Tools не входят в пользовательские требования. Перед выпуском каждой версии эта гарантия подтверждается чистым smoke test на Windows 11 x64 без C++ toolchain.

Docker, служба Windows, Task Scheduler, автозапуск, Electron/Tauri и иной desktop packaging в MVP не добавляются.

### Первый и обычный запуск

Контракт первого запуска будущей реализации:

```powershell
npm ci
npm run install:browsers
npm run build
npm run configure -- --telegram-executable 'C:\absolute\path\.venv\Scripts\telegram-alert.exe'
npm run doctor
npm start
```

`configure` принимает абсолютный путь к уже установленному и настроенному Telegram sender, проверяет его безопасной командой и транзакционно сохраняет как несекретную настройку в SQLite; отдельного config-файла нет. Команда не читает bot token: токен остаётся в Windows Credential Manager по контракту `telegram-alert-bus`.

Для обычного использования пользователь запускает `npm start` либо ярлык на поставляемый `start.cmd`. После успешной проверки среды, SQLite, Chromium и Telegram приложение слушает только `127.0.0.1:43117` и открывает `http://127.0.0.1:43117/` в браузере по умолчанию. Фиксированный loopback port служит single-instance барьером: повторный запуск, распознав уже работающий экземпляр по health endpoint и application identifier, только открывает его UI и завершается; чужой процесс на порту не останавливается, а даёт понятную ошибку.

Штатная остановка — `Ctrl+C` с ожиданием сообщения «Остановлено». Она использует уже выбранное восьмисекундное draining. Закрытие окна консоли остаётся аварийным, но восстанавливаемым сценарием, а не рекомендуемой операцией.

### Данные, диагностика и восстановление

Код не хранит изменяемое состояние рядом с репозиторием. Единый корень — `%LOCALAPPDATA%\WebsiteChangeMonitor`:

- `data\monitor.sqlite3` — единственный источник продуктовых данных и конфигурации, включая абсолютный путь Telegram executable;
- `backups\manual`, `backups\pre-update` и `backups\pre-migration` — проверенные копии БД;
- `logs` — ограниченные NDJSON-логи;
- `browsers` — восстанавливаемый Chromium Playwright.

Отдельного config-файла нет, а loopback port `43117` является фиксированной константой MVP. Bot token не попадает в SQLite и остаётся во внешнем Windows Credential Manager по контракту Telegram-модуля.

`npm run doctor` — read-only диагностика ОС/Node/npm, lockfile и native addon, каталогов и свободного места, версии/PRAGMA/integrity SQLite, Chromium, Telegram-контракта и порта. Повреждённая БД не заменяется пустой автоматически: пользователь получает путь к БД, последней проверенной копии и логу. Логи ротируются при 10 MiB, сохраняются последние 20 файлов и не содержат токен, сырой HTML, Снимки, diff, полный Telegram payload или секретные части URL.

Ручные `npm run backup` и `npm run restore -- --from '<absolute path>'` работают только при остановленном приложении. Backup создаётся SQLite backup API и считается готовым лишь после `quick_check`, `foreign_key_check` и проверки версии схемы. Restore сначала проверяет источник и сохраняет текущую БД, затем атомарно заменяет её через временный файл; старые `-wal/-shm` не подмешиваются. Ручные копии автоматически не удаляются; в каждом автоматическом классе — pre-update и pre-migration — сохраняются три последние проверенные копии.

### Безопасное обновление

Воспроизводимое обновление идёт не с движущегося `main`, а на явно указанную версионированную Git-tag. Поставляемый update script требует остановленного приложения и чистого worktree, запоминает исходный commit, создаёт проверенный pre-update backup, переключает checkout на точный tag, затем выполняет `npm ci`, установку браузера, build, тесты и `doctor --preflight`. Перед миграцией startup создаёт ещё одну проверенную pre-migration копию.

До миграции откат возвращает старый commit и его зависимости. После миграции откат всегда парный: **старый commit плюс соответствующая pre-update БД**; старый код на более новой схеме запускать нельзя. Telegram virtualenv обновляется независимо по правилам своего репозитория.

Критерий готовности релиза: весь первый запуск, повторный запуск, graceful stop, `doctor`, backup/restore, обновление и парный rollback проходят на чистой Windows 11 x64 машине с путём к репозиторию, содержащим пробелы и кириллицу.
