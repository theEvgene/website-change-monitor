# Windows release, update и rollback

Поддерживается foreground source-install на Windows 11 x64. Все команды выполняются при остановленном приложении и из корня Git checkout.

## Обновление на точный tag

Перед обновлением зафиксируйте или удалите локальные изменения. Команда принимает только существующий точный Git tag, а не ветку или произвольный commit:

```powershell
npm run build
node dist/server/cli.js update v0.2.0
```

Процесс проверяет чистоту checkout и tag, создаёт проверенный `pre-update` backup, переходит в detached HEAD на tag и последовательно выполняет `npm ci`, установку app-local Chromium, build, тесты и `doctor`. Коды doctor `0` и `2` разрешены; код `1` останавливает обновление.

После предварительных проверок создаётся отдельный `pre-migration` backup и только затем запускаются миграции. Для каждого автоматического вида сохраняются последние три копии. Файлы `manual-*.sqlite3` не удаляются.

При ошибке прежний commit и, если миграция уже началась, прежняя база восстанавливаются автоматически. Подробный редактированный журнал находится в `%LOCALAPPDATA%\WebsiteChangeMonitor\logs\application.ndjson`.

## Парный rollback

После успешного обновления приложение хранит указатель на прежний commit и соответствующий `pre-update` backup. Rollback разрешён только если checkout всё ещё соответствует последнему обновлению:

```powershell
node dist/server/cli.js rollback
```

Команда возвращает прежний код, заново устанавливает зависимости и Chromium, собирает его, восстанавливает соответствующую базу, запускает тесты и doctor. Откат только кода или только базы не предлагается.

## Release smoke

Автоматический smoke запускается из чистого профиля Windows 11 x64 в checkout, абсолютный путь которого содержит пробелы и кириллицу:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File scripts/release-smoke.ps1 -TargetTag v0.2.0
```

Smoke запускает собранное приложение, проверяет первый и повторный `start`, OpenAPI, настоящий HTTP preview, создание Монитора, Базовый снимок, Ручную проверку, Журнал, Сравнение, Telegram state и настройку браузерных уведомлений, после чего проходит тот же штатный путь остановки, что и обработчик сигнала. Полный набор интеграционных тестов дополнительно покрывает Изменение, автоматическую и Повторную проверки, UI/browser/Telegram states и update/rollback. Сам сценарий также создаёт и восстанавливает реальную SQLite-копию командами собранного приложения. Физическое нажатие `Ctrl+C` остаётся обязательным пунктом ручной части smoke, поскольку Windows не предоставляет надёжную переносимую отправку консольного события из дочернего Node-процесса.

После автоматической части вручную запустите приложение дважды и подтвердите:

1. Первый `start` открывает UI, второй переводит в уже работающий экземпляр; `Ctrl+C` корректно останавливает приложение.
2. Верхняя навигация содержит плотные таблицы, справа показана История выбранного Монитора.
3. Сравнение открывается широким модальным окном из контекста Проверки.
4. Недоступный Telegram показывает постоянную плашку и состояния доставки, но не блокирует Проверки.
5. Переключатель «Уведомлять при отсутствии изменений» сохраняется после перезапуска.
6. В UI выполняются preview, создание Монитора, baseline, Change, automatic/retry и просмотр Сравнения.

Результат smoke считается успешным только после автоматической части и этого ручного UI-чеклиста.
